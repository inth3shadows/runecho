Competitive positioning

RunEcho vs. the field:
prevention, not detection.

Every tool that beats RunEcho on accuracy only speaks after the code exists. RunEcho's job is to speak before it does. It says less — one narrow class of mistake, 4 of 9 in its own corpus — but it says it first, and it says it the same way every time.

The one thing it's built to do better

Gate an edit before it's written. The PreToolUse guard vets an Edit/Write inside the agent loop — blocking an invented symbol within a ~12 ms hook budget, with no build, no language server, and a deterministic, model-free receipt. Compilers, LSPs, and semantic MCP servers are all more accurate — and all run too late to stop the write.

agent proposes edit → RunEcho guard: block / ask → write lands → compiler / LSP / tests react

A search of MCP servers, IDE agents (Aider, Cline, Continue, Cody) and Claude Code hooks found no rival that deterministically blocks an undefined symbol pre-write. The one adjacent pre-write gate — GateGuard — only forces the agent to go look (run Grep/Read); it can't tell you a symbol doesn't exist. Cursor's hook surface can't host such a gate for its native edit tool: it exposes afterFileEdit but no pre-edit event. Verified against RunEcho's own code: liveIR re-parses a fresh IR on every call — no index to go stale.

The fair scorecard

Where it genuinely wins

  • Timing. The only tool that intervenes before the write, inside the agent loop.
  • Determinism. No LLM, no network — same input, same answer, every time. An auditable truth-trail receipt.
  • Never stale. Re-parse on every call; no index to fall behind the code.
  • Zero setup. One binary. No per-language server, no build, no API key.

Where it honestly loses

  • Accuracy of "exists?" A compiler or LSP is strictly more correct on any language it supports.
  • Depth. No types, no unexported/nested symbols, no receiver resolution — by design.
  • Semantic search. No embeddings, no "find code like this." Not its game.
  • Qualified refs. The guard skips obj.method() / pkg.Thing — a measured, deliberate gap.

Read before quoting 4/9 The corpus was mined from loud compiler/runtime errors (AttributeError, undefined type, TS2304), which inherently skew toward qualified and type positions. So 4/9 is a count, not a population rate — not "RunEcho catches 44% of all hallucinations." Every miss is a receiver-typed reference the guard abstains on by design (catching it needs type resolution RunEcho deliberately avoids). The direction is robust; the ratio isn't a percentage.

The read

Position RunEcho as the deterministic hallucination airbag for agent edits — not as a code-intelligence tool. Lean on what the metaphor already concedes: nobody buys an airbag believing it prevents crashes, and nobody resents it for that. It is one layer, priced and scoped like one, and it is deterministic about the moment it covers. Claiming more is what turns a tool people keep installed into one they turn off — and a guard that gets turned off protects nothing, which makes honesty a design constraint here rather than a virtue.

Framed as "code intelligence," it walks into a fight with Serena and every LSP and loses on depth. Framed as a pre-write, model-free gate, it stands where nothing else does: intervening before the mistake lands, across languages, with a receipt you can trust because no model produced it. The one adjacent pre-write gate — GateGuard — only forces the agent to go look; it can't tell you a symbol is undefined. That empty square is the whole position.

Concede depth on purpose. Against the compiler, win on timing + no build + language-agnostic. Against Serena, win on never-stale + zero setup. Lead with the bench/FINDINGS.md catch-rate corpus — measured evidence is the right weapon for a tool whose whole pitch is determinism.
